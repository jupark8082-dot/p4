# [PRD v2] AI 기반 발전소 운전 성능 예측 및 디지털 트윈 시각화 시스템

> **프로젝트명**: AI-Powered Power Plant Performance Predictor (P4)
> **버전**: v2.0 (2026-03-12)
> **변경 이력**: v1 리뷰 피드백 반영 — 데이터 지연 SLA, 예측 모델 운영 전략, DB 파티셔닝, 기술 스택 확정, 보안 설계, 로드맵 상세화

---

## 1. 프로젝트 개요

### 1.1. 목적
발전소의 OPC DA 실시간 운전/성능 데이터를 통합하고, 딥러닝 모델을 통해 미래 성능 지표를 예측하여 계통도 기반 웹 대시보드로 제공한다.

### 1.2. 핵심 가치
- 실시간 데이터와 예측값 비교를 통한 **선제적 운영 대응**
- 하이브리드 아키텍처(CS-Web)를 통한 **시스템 안정성 및 관리 편의성** 확보
- GPU 없는 서버 환경에서의 **효율적인 AI 학습/추론** 구현

### 1.3. 프로젝트 성격

> [!NOTE]
> 본 프로젝트는 발주 기반이 아닌 **자체 기획 프로젝트**입니다. 아래 명시된 기본값(Default)들은 합리적 가정에 기반하며, 실제 현장 적용 시 사용자 피드백에 따라 유연하게 조정될 수 있습니다. 각 섹션에서 `[기본값]` 표기가 이에 해당합니다.

## 3. 비기능 요구사항 (Non-Functional Requirements)

### 3.1. 데이터 갱신 SLA

| 항목 | 기본값 | 비고 |
|:---|:---|:---|
| OPC → DB 저장 지연 | `[기본값]` ≤ 2초 | CS Tool 내부 처리 |
| DB → Web 화면 갱신 | `[기본값]` ≤ 3초 | WebSocket 푸시 방식 채택 |
| End-to-End 지연 | `[기본값]` ≤ 5초 | OPC 발생 → 화면 표시 |

> [!TIP]
> 웹 대시보드는 **WebSocket**을 기본 통신 방식으로 채택하여 폴링 대비 지연 시간과 서버 부하를 동시에 줄인다. 초기 데이터 로딩 및 이력 조회는 REST API를 병행 사용한다.

### 3.2. 자원 최적화
- GPU 부재를 고려하여 **경량 신경망 구조** 채택 (Hidden Layer ≤ 3, Unit ≤ 128)
- 추론 시 **ONNX Runtime** 활용으로 CPU 환경 최적화
- 학습 시 **Incremental Learning(증분 학습)** 적용으로 전체 재학습 부담 감소

### 3.3. 가용성
- CS Tool 예외 처리: OPC 연결 끊김 시 **자동 재접속** (최대 5회, 지수 백오프)
- DB 연결 풀링 적용
- CS Tool 프로세스 **Windows 서비스** 등록으로 OS 부팅 시 자동 시작

### 3.4. 보안

| 영역 | 정책 |
|:---|:---|
| 네트워크 | 발전소 **폐쇄망** 운영 전제 |
| OPC DCOM | Windows 인증 + DCOM 보안 설정 |
| Web 인증 | `[기본값]` **JWT 토큰 기반** 자체 인증 (추후 AD 연동 확장 가능) |
| API 보호 | 모든 엔드포인트 JWT 인증 필수, 역할 기반 접근 제어 (Admin / Viewer) |
| 비밀번호 | bcrypt 해싱, 최소 8자 |

## 4. 기능 요구사항 (Functional Requirements)

### 4.1. [Admin CS Tool] 데이터 및 AI 관리

#### 4.1.1. OPC DA 인터페이스
- 서버 접속 설정 (호스트, ProgID) 및 구독(Subscription) 태그 관리
- **Deadband 로직**: 변화량이 설정 임계값 미만인 데이터는 필터링하여 DB 부하 감소
- 연결 상태 모니터링 및 자동 재접속

#### 4.1.2. 데이터 샘플링
- 실시간(초 단위) 데이터를 분석용(**`[기본값]` 1분 단위 평균**)으로 가공하여 `TB_HISTORY_MIN`에 저장
- 집계 주기는 설정 파일에서 변경 가능 (1분 / 5분 / 10분)

#### 4.1.3. 딥러닝 모델 빌더
- **인자 선택**: GUI를 통해 입력 변수(X)와 목적 변수(Y)를 태그 리스트에서 선택
- **알고리즘 선택**: `[기본값]` **LSTM** (RNN, GRU도 선택 가능)
- **하이퍼파라미터**: Epoch, Batch Size, Learning Rate, Sequence Length 등 GUI 조절
- **자동 학습 스케줄러**: `[기본값]` 매일 **02:00~05:00** 야간 시간대에 데이터셋 갱신 및 재학습

#### 4.1.4. 예측(추론) 파이프라인

| 항목 | 기본값 | 비고 |
|:---|:---|:---|
| **추론 주기** | `[기본값]` **5분마다** | 설정 가능 (1분~60분) |
| **예측 범위(Horizon)** | `[기본값]` **1시간 후** | 향후 확장 가능 (10분~24시간) |
| **입력 시퀀스 길이** | `[기본값]` **60 스텝** (= 과거 1시간) | 1분 단위 히스토리 기준 |
| **추론 엔진** | ONNX Runtime (CPU) | 학습은 PyTorch, 추론은 ONNX 변환 |

#### 4.1.5. 모델 라이프사이클 관리

```
학습 완료 → 검증(Validation RMSE) → 기존 모델 대비 비교
  ├─ 신규 모델 우수 → 배포(Active), 기존 모델 보관(Archive)
  └─ 신규 모델 열등 → 폐기(Discard), 기존 모델 유지
```

- 모델 버전은 **타임스탬프 기반** 자동 관리 (예: `model_lstm_tag001_20260312_0200.onnx`)
- `[기본값]` 최근 **5개 버전** 보관, 이전 버전은 자동 삭제
- 롤백: 관리자가 이전 버전을 수동 선택하여 Active로 변경 가능

#### 4.1.6. 데이터 Drift 감지

> [!NOTE]
> 발전소 데이터는 계절 변화, 정비 후 특성 변화 등으로 분포가 달라질 수 있다.

- `[기본값]` **PSI(Population Stability Index)** 기반 Drift 감지
- 주간 단위로 학습 데이터 분포 대비 최근 운전 데이터 분포 비교
- PSI > 0.2 시 관리자 알림 + 재학습 권고 플래그 활성화

### 4.2. [Web Dashboard] 실시간 모니터링 및 시각화

#### 4.2.1. PFD 오버레이 (Digital Twin)
- 발전소 전체 계통도(PFD) 이미지 위에 **동적 데이터 카드** 배치
- 데이터 카드 표시 항목:

| 항목 | 설명 | 색상 코드 |
|:---|:---|:---|
| **Actual** | 실시간 현재값 | 하얀색 (기본) |
| **Predicted** | AI 예측값 | 파란색 |
| **Deviation** | \|Actual - Predicted\| / Predicted × 100% | 초록(<5%), 노랑(5~10%), 빨강(>10%) |

#### 4.2.2. 트렌드 분석 (Trend)
- 과거 이력(History)과 미래 예측(Forecast) 데이터를 **단일 차트에 시각화**
- 현재 시점을 기준으로 좌측=실적, 우측=예측 (점선 구분)
- 태그별 **멀티 차트** 비교 기능 (최대 `[기본값]` 4개 동시)
- 조회 기간: `[기본값]` 과거 24시간 ~ 미래 1시간

#### 4.2.3. 사용자 정의 레이아웃
- 데이터 카드의 위치를 **드래그 앤 드롭**으로 배치
- 레이아웃 설정은 사용자 계정별 DB 저장
- 기본 레이아웃 템플릿 제공 (관리자가 설정)

#### 4.2.4. 알림 표시
- Deviation 임계값 초과 시 **화면 내 토스트 알림**
- 모델 Drift 감지 시 관리자 화면에 경고 배지 표시

## 5. 데이터베이스 설계

### 5.1. 테이블 구조

| 테이블 | 용도 | 보관 정책 | 파티셔닝 |
|:---|:---|:---|:---|
| `TB_REALTIME_DATA` | OPC 초 단위 데이터 | `[기본값]` 24시간 보관 후 일별 백업 테이블로 이동 | 일(Day) 단위 파티셔닝 |
| `TB_HISTORY_MIN` | 1분 단위 평균 데이터 | `[기본값]` 2년 보관 | 월(Month) 단위 파티셔닝 |
| `TB_PREDICT_RESULT` | 모델 예측 결과값 | `[기본값]` 1년 보관 | 월(Month) 단위 파티셔닝 |
| `TB_MODEL_INFO` | 모델 메타데이터 | 영구 보관 | — |
| `TB_USER` | 웹 사용자 계정/권한 | 영구 보관 | — |
| `TB_LAYOUT_CONFIG` | 사용자별 레이아웃 설정 | 영구 보관 | — |
| `TB_DRIFT_LOG` | Drift 감지 이력 | `[기본값]` 1년 보관 | — |

### 5.2. 데이터 볼륨 추정

> [!IMPORTANT]
> 아래는 태그 **500개** 기준 추정치이며, 실제 태그 수에 따라 선형 비례한다.

| 테이블 | 행/일 | 행/년 | 비고 |
|:---|:---|:---|:---|
| `TB_REALTIME_DATA` | ~4,320만 (Deadband 미적용) | 보관 안 함 | 24시간 롤링 |
| `TB_HISTORY_MIN` | 72만 | ~2.6억 | 2년 보관 시 ~5.2억 |
| `TB_PREDICT_RESULT` | ~14.4만 | ~5,256만 | 5분 주기 × 500태그 |

**파티셔닝 전략**: MS SQL의 **테이블 파티셔닝** 기능을 활용하여 월 단위 파티션을 생성하고, 보관 기한 초과 파티션은 `SWITCH` 명령으로 아카이브 테이블로 이동 후 삭제한다.

## 6. 기술 스택 (확정)

| 영역 | 기술 | 선정 사유 |
|:---|:---|:---|
| **CS Tool (Desktop)** | Python + PySide6 | DL 엔진과 동일 언어로 통합, OPC DA는 `OpenOPC` 또는 `opcda` 라이브러리 활용 |
| **Deep Learning** | **PyTorch** (학습) + **ONNX Runtime** (추론) | CPU 환경 추론 최적화, 모델 포터빌리티 확보 |
| **Database** | MS SQL Server 2019+ | 현장 표준, 테이블 파티셔닝 지원 |
| **Backend API** | **FastAPI** | 비동기 지원, WebSocket 내장, 자동 API 문서 생성 |
| **Frontend** | **React.js** + Plotly.js | 컴포넌트 기반 UI, 대화형 차트 라이브러리 |
| **통신** | WebSocket (실시간) + REST (이력 조회) | 지연 시간 최소화 + 범용성 병행 |

| Deviation 알림 | 오차 임계값 초과 시 화면 알림 표시 |
| Drift 감지 | PSI 기반 분포 변화 감지 + 재학습 권고 |
| 야간 자동 학습 | 스케줄러 기반 야간 재학습 자동 수행 |
| 통합 테스트 | OPC → DB → AI → Web 전 구간 E2E 검증 |
| **마일스톤** | ✅ 전체 파이프라인이 24시간 무중단 운영됨을 확인 |

## 8. 기본값 변경 가이드

본 문서에서 `[기본값]`으로 표기된 항목들은 아래 설정 파일에서 일괄 관리한다:

```yaml
# config/defaults.yaml (예시)
data:
  sampling_interval_min: 1        # 분 단위 집계 주기
  realtime_retention_hours: 24    # 실시간 데이터 보관 시간
  history_retention_years: 2      # 히스토리 보관 기간

prediction:
  inference_interval_min: 5       # 추론 주기 (분)
  forecast_horizon_min: 60        # 예측 범위 (분)
  input_sequence_length: 60       # 입력 시퀀스 길이 (스텝)

model:
  default_algorithm: LSTM         # 기본 알고리즘
  max_versions: 5                 # 보관 모델 버전 수
  training_schedule: "0 2 * * *"  # cron 표현식 (매일 02:00)

drift:
  method: PSI                     # Drift 감지 방법
  threshold: 0.2                  # PSI 임계값
  check_interval_days: 7          # 검사 주기

web:
  max_trend_charts: 4             # 동시 비교 차트 수
  deviation_warn_pct: 5           # 노랑 경고 (%)
  deviation_alert_pct: 10         # 빨강 경고 (%)
```

> [!TIP]
> 현장 적용 시 위 설정값만 조정하면 코드 수정 없이 시스템 동작을 변경할 수 있다.
